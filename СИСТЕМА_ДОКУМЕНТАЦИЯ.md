# ДОКУМЕНТАЦИЯ СИСТЕМЫ СЛЕЖЕНИЯ ЗА ВАГОНАМИ
## Полное описание для презентации

---

## 1. ЧТО ЭТО ТАКОЕ — В ДВУХ СЛОВАХ

Система автоматически отслеживает движение железнодорожных вагонов, которые едут **на наш завод или с нашего завода** (станция Круглое Поле, код **648400**). Она получает данные из внешней системы дислокации, анализирует их, понимает — кто из вагонов едет к нам, кто уже приехал, кто уехал — и предоставляет удобный интерфейс для управления приёмкой груза и передачи данных в 1С.

## 3. ОТКУДА БЕРУТСЯ ДАННЫЕ

### Таблица `dislocation` — сырые данные
Это «сырой поток» от внешней системы слежения. Каждая запись — одно событие с вагоном:
- **railway_carriage_number** — номер вагона (8 цифр)
- **flight_start_date** — дата/время начала рейса
- **flight_start_station_code** — станция отправления рейса
- **destination_station_code** — станция назначения
- **station_code_performing_operation** — где прямо сейчас произошла операция
- **operation_code_railway_carriage** — код операции (96=выгружен, 20=прибыл, 80=передача и т.д.)
- **remaining_distance** — остаток пути в км
- **date_time_of_operation** — когда произошла операция
- **number_train** — номер поезда
- **waybill_number** — номер накладной
- **flight_id** — ← это поле мы ЗАПОЛНЯЕМ сами, привязывая к нашему рейсу

### Таблица `etran_waybills` — накладные ЭТРАН
Данные из системы ЭТРАН (электронные транспортные накладные ГУ-27):
- Накладная = один договор перевозки
- Один вагон может ехать по нескольким накладным (разные партии)
- Одна накладная может содержать несколько вагонов
- К вагону могут быть привязаны контейнеры (КТК)

## 4. КАК РАБОТАЕТ СИНХРОНИЗАЦИЯ — ШАГИ ПОДРОБНО

Синхронизация запускается **каждые 10 минут** автоматически (планировщик) или вручную по кнопке «Обновить данные».

### ШАГ 0 — Отбор нужных вагонов (квалификация)

Из всего потока дислокации берём только те вагоны, которые **имеют отношение к станции 648400 (Круглое Поле)**:

```
Вагон попадает в систему если хотя бы одно условие верно:
  - Станция текущей операции = 648400
  - Станция назначения = 648400
  - Станция содержит '648400' (LIKE '%648400')
```

Берём **последнюю операцию** на каждую комбинацию (вагон + дата рейса).

**Защитный механизм:** Если вдруг фильтр вернул 0 строк (сбой данных, технические работы) — система **не архивирует** существующие активные рейсы, а берёт все последние операции без фильтра. Это защита от ложного массового архивирования.

### ШАГИ 1-2 — Создание вагонов и рейсов

Для каждого квалифицированного вагона система:
1. Находит или создаёт запись в таблице **wagons** (один вагон = одна запись на весь срок жизни)
2. Находит или создаёт **wagon_trip** (рейс) с ключом:
   - Номер вагона
   - Бизнес-дата начала рейса (в московском времени UTC+3, точность до дня)
   - Код станции отправления

> **Почему бизнес-дата?** Источник данных иногда присылает одну и ту же дату с микросекундной разницей. Мы нормализуем до дня — это предотвращает создание дублей.

### ШАГ 3 — Привязка dislocation.flight_id

Всем строкам в таблице `dislocation`, у которых ещё нет `flight_id`, присваиваем ID найденного или созданного рейса. Это связывает сырое событие с нашим рейсом.

### ШАГ 3b — Слияние дублей рейсов

Если по какой-то причине для одного (вагон + дата + станция) создалось два рейса — удаляем лишний, переносим комментарии и историю на основной.

### ШАГ 4 — Заполнение станции отправления

Для рейсов где название станции отправления пустое — заполняем из первой операции этого рейса.

### ШАГ 5 — Обновление полей рейса + расчёт активности

Это главный шаг. Для каждого рейса берём **самую позднюю операцию** и обновляем:
- Последняя дата операции, код, название
- Текущая станция (код и название)
- Станция назначения
- Номер и индекс поезда
- Остаток пути в км
- Номер накладной

И сразу же рассчитываем **is_active** (активен ли рейс) по правилам архивирования (см. раздел 5).

### ШАГ 5b — Дозаполнение номера накладной

Если в последней операции номер накладной пустой, ищем его в других операциях этого же рейса.

### ШАГ 5c — Пересборка связей рейс ↔ накладная

Для всех активных рейсов пересобираем таблицу `trip_waybills`:
- Берём номер вагона
- Находим накладные ЭТРАН, где этот вагон упомянут
- Создаём связи

### ШАГ 6 — Присвоение номеров рейсов

Рейсы без номера (`flight_number IS NULL`) нумеруются последовательно, начиная от текущего максимума. Номера глобально уникальны по всей системе (как номер документа в 1С).

### ШАГ 7 — Обновление активности вагонов

Вагон активен ↔ у него есть хотя бы один активный рейс.

### ШАГ 7b — Реактивация архивных рейсов

Иногда бывает так: пришли данные с ошибочной датой рейса → система создала новый рейс, старый ушёл в архив. Потом данные исправились и операции снова пришли на старый рейс. Если:
- Рейс в архиве
- Его `last_operation_date` **свежее**, чем у всех других рейсов этого вагона
- И у него нет терминального кода операции (не завершён)

→ Реактивируем рейс. Затем шаг 8 выберет правильного победителя.

### ШАГ 8 — Нормализация: один активный рейс на вагон

У каждого вагона не может быть больше одного активного рейса. Если активных несколько — оставляем тот, у которого **самая свежая last_operation_date**. Остальные архивируем.

### ШАГ 8b — Архивирование устаревших рейсов

Если у вагона есть активный рейс, но при этом есть **другой рейс** (неважно, активный или архивный) с более свежей `last_operation_date` **и терминальным кодом** — архивируем активный рейс. Смысл: вагон уже завершил маршрут, незачем держать старый рейс активным.

## 5. ПРАВИЛА АРХИВИРОВАНИЯ РЕЙСОВ

Рейс уходит в архив (is_active = false) если выполнено **хотя бы одно** условие:

| Условие | Что означает |
|---------|--------------|
| Код операции = **96** | Вагон сдан государству / выгружен |
| Код операции = **85** | Специальный код завершения |
| Станция операции = **628406** | Вагон прибыл на конечную базу |
| Остаток = 0 + код = **20** | Прибыл на станцию назначения |
| Остаток = 0 + код = **43, 58, 81** | Технические операции завершения (промывка, ремонт, перегрузка) |
| Отправился с **648400**, едет НЕ в 648400, остаток = 0 | Наш вагон завершил рейс «туда» (отправка с завода) |
| Код = **80** + станция операции = станция назначения | Передача на другую ж/д, прибыл в конечную |
| Станция операции = **648400** + назначение = **648400** + последняя операция > 2 дней назад | Вагон приехал к нам, стоит >2 суток — считаем принятым |

## 7. БИЗНЕС-ПРОЦЕСС — КАК ПРИЁМКА РАБОТАЕТ НА ПРАКТИКЕ

### Этап 1: Мониторинг (вагон далеко)
- Вагон едет к нам, остаток > 150 км
- Система отслеживает его, показывает в **Дислокации** (вкладка «Поставка»)
- Вкладка «Поезда» показывает поезд в статусе **«Мониторинг»**
- Можно раскрыть и посмотреть состав, но управлять заявками нельзя

### Этап 2: Формирование болванки (вагон близко)
- Остаток стал ≤ 150 км
- **Автоматически** создаётся «болванка» маршрута (`railway_route`)
- Делается снимок (snapshot) состава поезда: кто едет, с какими накладными и КТК
- Вкладка «Поезда» показывает поезд в статусе **«Формируется»**

### Этап 3: Управление заявками
- Диспетчер/логист открывает вкладку **«Поезда»**, раскрывает поезд
- Видит таблицу: вагон → накладная → КТК → отправитель → получатель → груз
- Выбирает нужные строки (галочками) и нажимает **«Назначить клиентов»**
- Заполняет: клиент, комментарий
- Создаётся **Заявка** (receiving_order) с привязанными строками

> Один поезд может иметь несколько заявок (разные клиенты/партии).
> Одна строка (накладная+вагон+КТК) не может быть в двух заявках одновременно.

### Этап 4: Экспорт в 1С
- Нажимает **«Сформировать JSON»**
- Система отдаёт JSON со всеми заявками, вагонами, накладными, КТК
- JSON передаётся в 1С для оформления приёмки
- Маршрут автоматически переходит в статус **«Закрыт»**

### Этап 5: Архивирование
- После приёмки операции с кодами завершения (96, 20 и т.д.) приходят из источника
- Синхронизация архивирует рейсы
- Вагоны пропадают из активных списков

## 8. ИНТЕРФЕЙС — ВКЛАДКИ ПОДРОБНО

### Вкладка «Дислокация»

**Что показывает:** Все вагоны под слежением + их последние операции

**Фильтры:**
- **Поставка** — вагоны с назначением на 648400 (едут к нам)
- **Отправка** — вагоны с отправлением с 648400 (уехали от нас)
- **Архивные** — завершённые рейсы
- **Все** — полная картина

**Возможности:**
- Поиск по номеру вагона (можно вводить несколько через пробел)
- Сортировка по дате последней операции
- Группировка по поезду / по диапазону остатка км
- Массовое назначение комментариев
- Раскрытие вагона → список рейсов → история операций по рейсу

**Столбцы:** Номер вагона, текущая станция, операция, дата операции, остаток км, поезд, накладная, клиент, комментарий

### Вкладка «Рейсы»

**Что показывает:** Список всех рейсов (wagon_trips) с детальной информацией

**Фильтры:** Активные / Архивные / Все

**Каждая строка:** Номер рейса, вагон, маршрут (откуда→куда), дата начала, последняя операция, остаток, поезд

**Раскрытие:** История всех операций по рейсу в хронологическом порядке

### Вкладка «Поезда»

**Что показывает:** Поезда, едущие с грузом на 648400 (только поставки)

**Статусы поезда:**
- **Мониторинг** — далеко (> 150 км), только просмотр
- **Формируется** — ≤ 150 км, болванка создаётся
- **Открыт** — болванка создана, заявки можно редактировать
- **Закрыт** — после экспорта в 1С, только просмотр

**Колонки:** № поезда, индекс, кол-во вагонов, кол-во с накладной, мин. остаток, статус

**Раскрытие поезда (Мониторинг):**
- Таблица состава: вагон, накладная, КТК, отправитель, получатель, груз, остаток, последняя операция
- Управление заявками недоступно

**Раскрытие поезда (Открыт):**
- Та же таблица, плюс колонка «Клиент» (назначенная заявка)
- Кнопка «Назначить клиентов» — выбор строк и создание заявки
- Цветная подсветка заявок (каждая заявка — свой цвет)
- Кнопка «Сформировать JSON» — экспорт в 1С
- Редактирование / удаление заявок

### Вкладка «Рейсы в ЛКДС» (только администратор)

**Что показывает:** Все когда-либо созданные маршруты (болванки)

**Колонки:** № поезда, индекс, статус (Открыт/Закрыт), кол-во заявок, кол-во вагонов, дата создания

**Раскрытие маршрута:** Вагоны, сгруппированные по заявкам:
- Заголовок группы: № заявки + клиент
- Таблица: вагон, накладная, КТК, отправитель, получатель, груз

### Вкладка «Заявки» (только администратор)

**Что показывает:** Все заявки во всей системе

**Колонки:** № заявки, клиент, № поезда, индекс, статус рейса, кол-во вагонов, комментарий, дата создания

**Раскрытие заявки:** Полная таблица вагонов/КТК с деталями

## 9. ТЕХНИЧЕСКИЕ ДЕТАЛИ (если спросят)

### Как работает уникальность строки в заявке

Один и тот же вагон может ехать с несколькими накладными (разные грузы). Поэтому ключ заявки — это **тройка**:
1. Накладная (waybill_id) — если есть
2. Контейнер (container_number) — если есть КТК
3. Вагон (wagon_number) — всегда есть

Если накладная + КТК → уникальность по ним.
Если накладная, нет КТК → уникальность по накладной + вагону.
Если нет накладной → уникальность по вагону.

Это реализовано через partial unique indexes в PostgreSQL.

### Как мы знаем что вагон в нашем поезде

Когда вагон передвигается, в `dislocation` приходит операция с `number_train`. Это поле сохраняется в `wagon_trips.number_train`. По этому полю группируем вагоны в «поезда» в интерфейсе.

### Почему заявка привязана к конкретному рейсу (wagon_trip_id)

Один вагон за год может сделать 50+ рейсов. Привязка к конкретному `wagon_trip_id` гарантирует, что заявка «помнит» именно этот конкретный рейс вагона. Если вагон переедет на другой поезд (что бывает — перицепка состава), система всё равно правильно найдёт его заявку, потому что рейс тот же, только `number_train` изменился.

### Болванка и снапшот

Когда остаток ≤ 150 км, система делает «снимок» (snapshot) состава поезда. Это JSONB в `railway_routes`. Он нужен как страховка — если потом вагоны уйдут в архив, мы всё равно знаем каким был состав в момент приёмки. Но при открытом маршруте всегда показываем **живые данные** (актуальнее снапшота).

### Синхронизация раз в 10 минут

Планировщик (APScheduler) запускает `sync_all()` каждые 10 минут:
1. Синхронизация старой модели (`tracking_wagons`) — для обратной совместимости
2. Синхронизация новой иерархической модели (вагоны → рейсы)
3. Проверка и создание болванок для поездов ≤ 150 км

### Кнопка «Обновить данные»

Делает то же самое что автоматический синк, но прямо сейчас. Занимает 10-30 секунд в зависимости от объёма данных.

## 10. ТИПИЧНЫЕ ВОПРОСЫ И ОТВЕТЫ

**В: Откуда вы знаете что вагон едет к вам?**
О: Источник данных содержит поле «станция назначения». Если там указан код 648400 (наша станция Круглое Поле) — вагон едёт к нам. Плюс отслеживаем вагоны, у которых станция текущей операции тоже 648400 — значит уже приехали.

**В: Что если данные задержатся или придут с ошибкой?**
О: Несколько защит:
1. Если источник прислал 0 вагонов — система ничего не архивирует (fail-safe)
2. Если вагон пропал из дислокации, но не получил терминальный код — остаётся активным до 20 дней (старая модель)
3. Если данные пришли с ошибочной датой рейса — система потом исправляет сама (Step 7b реактивации)

**В: Можно ли смотреть историю операций?**
О: Да. В вкладке «Дислокация» раскрываешь вагон → раскрываешь рейс → видишь все операции по хронологии.

**В: Кто может создавать/изменять заявки?**
О: Любой авторизованный пользователь может создавать заявки во вкладке «Поезда». Просмотр всех заявок и маршрутов (вкладки «Заявки» и «Рейсы в ЛКДС») — только администраторы.

**В: Что происходит когда вагоны разных клиентов едут в одном поезде?**
О: Создаём отдельную заявку для каждого клиента в рамках одного поезда. Каждая заявка — свой клиент, свои вагоны/КТК. В интерфейсе каждая заявка подсвечена своим цветом.

**В: Что если один вагон везёт груз двух клиентов (два контейнера)?**
О: Такой вагон появится в таблице как **две строки** — по одной на каждый контейнер (КТК). Каждый контейнер можно включить в разные заявки разным клиентам.

**В: Что происходит после экспорта JSON?**
О: Маршрут переходит в статус «Закрыт» — заявки больше нельзя редактировать. JSON передаётся в 1С для оформления документов приёмки.

**В: Сколько вагонов система может отслеживать?**
О: Ограничений нет. Синхронизация батчевая (SQL UPDATE), не построчная — одинаково хорошо работает на 100 и на 10 000 вагонах.

**В: Что случится если вагон отцепят от одного поезда и прицепят к другому?**
О: В источнике данных придёт операция с новым `number_train`. На следующем синке `wagon_trips.number_train` обновится. Заявка на вагон сохранится — она привязана к `wagon_trip_id` (рейсу), а не к номеру поезда.

**В: Система работает в реальном времени?**
О: Данные обновляются каждые 10 минут автоматически. Можно обновить вручную в любой момент кнопкой «Обновить данные». Задержка — от нескольких секунд до 10 минут относительно источника.

## 12. КРАТКАЯ ШПАРГАЛКА ДЛЯ ПРЕЗЕНТАЦИИ

```
ИСТОЧНИК ДАННЫХ → каждые 10 мин → СИНХРОНИЗАЦИЯ → БАЗА ДАННЫХ → ИНТЕРФЕЙС

Фильтр по станции 648400 (Круглое Поле) — берём только наши вагоны

Иерархия:
  Вагон → много Рейсов → много Операций
  Рейс → Накладные ЭТРАН → Контейнеры (КТК)

Поток приёмки:
  Мониторинг (>150 км) → Болванка создана (≤150 км) → Заявки → Экспорт в 1С → Закрыт

Архивирование рейса: код 96, 20+rem=0, 80+приехал в нпзн, >2 дней в 648400

Ключ уникальности строки заявки: накладная + КТК + вагон (в зависимости от наличия)

Защита от ошибок источника:
  - Fail-safe при 0 результатах (не архивируем)
  - Реактивация рейсов при исправлении данных
  - wagon_trip_id — точная привязка даже при смене поезда
```
